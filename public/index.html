More detail

Engineered a production data pipeline that ingests live market feeds, propagates events in real-time via PostgreSQL LISTEN/NOTIFY (zero-polling architecture), and delivers updates to connected trading clients through Socket.IO. Handles 500+ concurrent WebSocket connections with consistent sub-100ms end-to-end latency, automatic reconnection with exponential backoff, and server-side backpressure management.

Architecture: Market data feed → Node.js ingestion layer → PostgreSQL (INSERT + NOTIFY trigger) → LISTEN channel handler → Socket.IO broadcast → connected trading clients. No polling at any layer — entirely event-driven from database to browser.

Performance: P95 latency <85ms from feed ingestion to client delivery. Tested at 500+ simultaneous WebSocket connections with zero message drops. Connection pool management via pg-pool with configurable max connections and idle timeout.

Challenge solved: PostgreSQL LISTEN/NOTIFY has no built-in delivery guarantee. Implemented a sequence-number-based reconciliation system where clients can request missed events on reconnection, ensuring eventual consistency without adding a message queue dependency.

GitHub

Offline-first respiratory diagnostics — 3-model ensemble for TB/Asthma screening, designed for clinics with zero internet.

PyTorch 2.0 TensorFlow 2.x FastAPI React 18 TypeScript librosa Recharts
More detail

Three-model ensemble (PANNs CNN14 for robust feature extraction + YAMNet for fast anomaly detection + custom CNN-BiLSTM for targeted respiratory classification) analyzes cough and breathing audio for TB and Asthma risk prediction. Runs entirely on-device — complete patient data privacy with zero network requests. Includes multilingual triage agent supporting Hindi, Gujarati, Marathi, and English, plus dynamic outbreak alerting based on spatial case clustering.

Architecture: Audio capture (16kHz WAV) → mel-spectrogram extraction via librosa → parallel inference across 3 models → weighted ensemble voting (0.4 PANNs + 0.3 YAMNet + 0.3 CNN-BiLSTM) → risk classification → triage Q&A agent → localStorage patient records → outbreak heatmap. The ensemble cross-validates predictions: PANNs provides robust general audio understanding, YAMNet catches anomalies the others miss, and the custom CNN-BiLSTM is fine-tuned specifically on respiratory audio. Combined accuracy: 91% on validation set with <3% false negative rate on TB indicators.

Challenge solved: Designing for clinics with zero internet and health workers who may not speak English. The multilingual triage agent generates follow-up questions in the patient's language, and all patient records persist locally via localStorage — no server required for data continuity.

VoteChain

GitHub

Blockchain e-voting with SHA-256 chain validation, Aadhaar-based voter auth, designed for Maharashtra state elections.

Java 17 Spark Java MongoDB SHA-256 Blockchain
More detail

Tamper-proof electronic voting platform using a custom blockchain implementation for vote integrity — each vote is a block with SHA-256 hash linking and Merkle tree verification. MongoDB handles voter registration with Aadhaar-based authentication (simulated). Real-time vote tallying with automatic chain validation on every new block addition.

Architecture: Voter registration (MongoDB + Aadhaar lookup) → authentication → vote casting → block creation (vote data + previous hash + timestamp + nonce) → SHA-256 hash computation → chain append with Merkle root update → real-time tally aggregation → admin dashboard.

Why blockchain here: Most "blockchain projects" are solutions looking for a problem. E-voting is one of the few domains where blockchain's core property — immutable, append-only audit trails — is architecturally essential. The chain provides tamper-evidence: any modification to a past vote invalidates all subsequent hashes, making fraud detectable in O(n) time.

Challenge solved: Balancing vote anonymity with auditability. Votes are stored as hashed representations — the chain proves a vote was cast and unchanged, but the voter's identity is decoupled from their ballot through a two-phase commit protocol.

NyayaAI — Legal AI Assistant

GitHub

RAG-based legal AI for Indian jurisprudence — hybrid retrieval combining local knowledge base + live web search. In revamp.

Python 3.11 LangChain RAG Flask ChromaDB Sentence-Transformers
More detail

Context-aware legal Q&A system combining a local vector knowledge base (ChromaDB with sentence-transformer embeddings) with real-time web search for comprehensive Indian legal information. Uses RAG pipeline with LangChain orchestration — queries first hit the local corpus of Indian legal documents, then augment with live web results for currency. Every answer includes source citations with relevance scores.

Architecture: User query → query embedding (sentence-transformers all-MiniLM-L6-v2) → ChromaDB similarity search (top-k=5, cosine distance) → web search augmentation (SerpAPI fallback) → context assembly with source deduplication → LLM generation with citation formatting → response with inline source links and confidence scores.

Why hybrid retrieval: Legal information has a freshness problem — statutes change, new judgments are delivered, amendments are passed. A pure local corpus goes stale. The hybrid approach grounds answers in curated legal texts (accuracy) while supplementing with current web data (currency). Retrieval precision: 78% on Indian legal Q&A benchmark.

Status: Currently in revamp — migrating from Flask to FastAPI, upgrading embeddings to legal-domain-specific models, and expanding the corpus to cover recent Supreme Court judgments through 2025.
